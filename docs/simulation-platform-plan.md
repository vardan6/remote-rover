# Simulation Platform Plan

## Purpose

This document defines the transition plan for the next simulator platform under the `remote-rover` repository.

It replaces the earlier generic simulation-improvement framing with a concrete parallel-successor plan:
- keep the current `3d-env` working during transition
- introduce a new simulator sub-project beside it
- keep `gcs_server` as the main operator-facing application
- preserve compatibility with the current MQTT/GCS contract while the new simulator is built out

This document is not a replacement for `docs/implementation-roadmap.md`.
It is a companion plan focused on simulator evolution, asset workflows, map support, and logging/replay.

## Implementation Status

The plan is now partially implemented.

Implemented from this plan:
- config-backed backend identity in the GCS
- `SQLite`-backed GCS replay/session storage
- GCS replay APIs for session list, session detail, and session rollover
- separate GCS replay page
- first Leaflet-based replay map
- normalized GCS-side telemetry handling for replay storage
- `rover-sim-next/` repository scaffold with ROS 2/Gazebo-oriented structure

Not implemented yet from this plan:
- simulator-side logging in `3d-env`
- live map panel on the main GCS dashboard
- working `rover-sim-next` simulator backend
- authoritative CAD/asset import pipeline
- synchronized recorded video playback

## Current Implementation Reality

The current simulator is a working Panda3D desktop application with Bullet physics.
It already integrates correctly with the Ground Control Station over MQTT and supports the end-to-end operator loop.

Implemented today:
- custom Panda3D runtime and scene
- Bullet heightfield terrain
- custom `BulletVehicle` rover model
- procedural terrain and world geometry
- MQTT control subscription
- MQTT telemetry publication
- MQTT camera-frame publication
- GCS presence-aware publish gating
- pseudo-GPS derived from local XY coordinates

Not implemented yet:
- browser live map widget on the main dashboard
- proper geospatial coordinate transform model beyond current pseudo-GPS compatibility
- CAD-driven asset pipeline
- authoritative asset replacement workflow
- simulator-side logging architecture
- synchronized video recording and playback

The current simulator is good enough for demos, operator workflow development, MQTT integration, and iterative UI work.
It is not yet the right long-term basis for modular engineering assets, replay workflows, and a future simulator replacement path.

## Transition Goal

Create a new simulator sub-project that runs in parallel with `3d-env`, works with the existing `gcs_server`, and eventually replaces `3d-env`.

The transition goals are:
- avoid breaking the current prototype
- preserve the current MQTT/GCS integration contract
- support both simulators during migration
- add map support early in the GCS
- add structured session logging and replay from the GCS
- allow early prompt-driven asset generation
- allow later replacement by professional CAD-derived assets

## Recommended New Sub-Project Name

Recommended working name:
- `rover-sim-next`

Good alternatives:
- `rover-sim`
- `mission-sim`
- `sim-next`

Why `rover-sim-next` is the current recommendation:
- explicit that it is the successor to `3d-env`
- intuitive in the repository
- avoids locking the final long-term product name too early

## Current Technical Direction

The current repository direction for the successor simulator is now:
- long-term architecture target: `ROS 2 + Gazebo`
- current implementation state: scaffold only
- contract strategy: keep MQTT/GCS compatibility with `3d-env` during transition

This means the engine direction is no longer purely abstract.
It is chosen at the repository-structure level, even though the real simulator backend is still to be implemented.

## Compatibility Contract With `gcs_server`

The new simulator must preserve the current minimum MQTT compatibility surface so the existing GCS can work with either simulator during transition.

### MQTT Topics To Preserve

- `{topic_prefix}/{control_topic}`
- `{topic_prefix}/{state_topic}`
- `{topic_prefix}/{camera_topic}`
- `{topic_prefix}/{gcs_presence_topic}/{gcs_id}`

### Minimum Telemetry Fields To Preserve

- `timestamp`
- `position`
- `gps`
- `orientation.heading_deg`
- `speed`
- `camera.mode`
- `camera.video_endpoint`
- `power`

The new simulator may add more fields, but it must not break the live GCS dashboard or the replay normalization layer.

### Transition Rule

During migration:
- the GCS must be able to operate with either `3d-env` or `rover-sim-next`
- one active simulator backend is expected at a time from the GCS point of view
- backend identity must be explicit in config or runtime selection

## Asset Strategy

The simulator plan must support two classes of assets from the start.

### 1. Development Assets

Development assets are allowed and expected in the early phases.

They may be:
- generated by Codex
- generated procedurally in code
- quickly modeled for iteration

They are valid for:
- rover body and wheel geometry
- terrain props
- buildings
- obstacles
- map/site objects
- early world layout

They are temporary development artifacts and must not be treated as engineering source-of-truth.

### 2. Authoritative Assets

Authoritative assets are the later professional replacements.

They come from:
- real CAD source files
- validated dimensions
- engineering drawings
- approved exported simulation meshes

They become the source of truth for geometry and dimensions.

### Asset Replacement Requirement

The simulator and GCS-facing behavior must not depend on whether an asset is development or authoritative.
Authoritative assets must be able to replace development assets without changing:
- MQTT topic contracts
- replay model
- GCS behavior

## CAD And Mesh Workflow

Each rover or environment object should eventually be maintainable independently as an engineering asset.

### Required Asset Record

Each significant object should have:
- asset id
- asset class: development or authoritative
- source reference
- visual mesh path
- collision mesh path
- units and scale metadata
- optional physical properties
- optional drawing reference

### Workflow

1. Create or generate an early development asset for fast iteration.
2. Use it in the simulator to build features and validate workflows.
3. Later replace it with authoritative CAD-derived geometry.
4. Export simulation-ready visual and collision meshes separately.
5. Keep provenance metadata so replacements are controlled and traceable.

### Mesh Rules

- keep visual and collision meshes separate
- use `glTF`/`GLB` as the preferred simulation-ready mesh interchange format
- use much simpler collision meshes than visual meshes
- plan convex decomposition for non-trivial collision shapes
- do not use the simulator scene as the authoritative CAD source

### CAD Tooling Defaults

The plan should name default tooling so the workflow is actionable.

Recommended defaults:
- `FreeCAD` as the default open CAD authoring tool for authoritative engineering assets
- `OpenSCAD` as an optional tool for parametric development parts and quick procedural fixtures
- `Blender` for mesh cleanup, decimation, UV/material work, and export preparation

Commercial CAD tools are still valid if the engineering team already uses them, but they should be treated as interchangeable source systems feeding the same simulation asset pipeline.

## Map And Coordinate Model

Map support is an early GCS feature, not a late enhancement.
It must work with both `3d-env` and `rover-sim-next`.

### Immediate Reality

Today the simulator emits:
- local position
- heading
- pseudo-GPS values

The GCS shows GPS text values but does not yet render an actual map.

### Required Coordinate Model

The future telemetry model must define:
- explicit local coordinate convention
- configurable site origin
- deterministic transformation from local rover coordinates to map coordinates
- one normalized map/position model shared by live view and replay

Recommended local convention:
- ENU-style local frame

Recommended transform stack:
- `pyproj` for coordinate transforms
- `UTM` as the default practical projection for field deployments
- local tangent plane conversion as the high-accuracy small-site option when needed

Current pseudo-GPS should be treated as a temporary compatibility mechanism only.

### Map Library Decision

The first map implementation in the GCS should be:
- `Leaflet.js` for the first live/replay map

Reasons:
- fits the current browser UI well
- simple to integrate beside the existing dashboard and future replay page
- sufficient for 2D site maps, rover markers, heading arrows, and breadcrumb tracks

Future upgrades remain open:
- `MapLibre GL JS` if vector tiles or more advanced map motion become necessary
- `CesiumJS` only if a true 3D globe or terrain view becomes a real requirement

### Map Planning Rule

The live map and replay map must use the same normalized track model so historical playback and live operation remain semantically consistent.

## Logging And Replay Architecture

Logging and replay are first-class architecture requirements.
They are not optional future add-ons.

### Required Capabilities

- logging must be possible in both simulator and GCS
- replay visualization must be available from the GCS
- replay must support logs recorded from either origin
- the replay architecture must be designed now so synchronized video can be added later without redesign

### First Storage Backend

Use `SQLite` first.

Why:
- simple local deployment
- structured querying
- easy session-based replay
- suitable for both simulator-side and GCS-side capture
- can later be migrated to PostgreSQL if needed

### First Logging Scope

The first replay-capable capture scope should include:
- telemetry
- control frames
- runtime and broker events

Video recording can come later, but the replay session model must reserve synchronization support for it now.

### Required Session Concepts

Each recorded session should capture:
- session id
- source node id
- backend type: `3d-env`, `rover-sim-next`, or `gcs`
- start and end timestamps
- backend version
- site/map context
- capture capability metadata

To preserve a clean path to future synchronized video, the session model should also reserve:
- frame timestamp or presentation timestamp (`pts`) for camera-related events
- optional camera frame index when raw frame streams are captured
- optional external media reference so video files can live outside the main SQLite database while still remaining attached to a session id

### Data To Record In Phase 1

- telemetry payloads as emitted or observed
- control payloads as sent or received
- GCS presence state changes
- controller ownership changes
- broker connect, disconnect, and reconnect events
- runtime warnings and errors relevant to replay interpretation
- camera frame timing metadata for later video synchronization

### Logging Rule

Simulator-side logging records what the backend produced and received locally.
GCS-side logging records what the GCS observed, normalized, and emitted.

Replay in the GCS must be able to consume logs from either origin.

## Replay UX In The GCS

Replay should be implemented first as a separate page in the GCS.

Why:
- keeps the live dashboard focused on operation
- avoids overloading the main page with replay controls
- gives a clean place for timeline, map, and future video synchronization

### First Replay Page Capabilities

- session list and session selection
- load one session into a replay controller
- map playback of rover movement
- telemetry value playback
- control-state timeline
- runtime event markers
- session clock with play, pause, seek, and speed control
- clear display of recording origin: simulator or GCS

### Future Video Capability

The replay page should reserve a video panel or playback slot even before recorded video is fully implemented.
This avoids redesigning the page when synchronized video is added later.

The first replay schema should assume:
- recorded video will eventually be stored as a separate media file referenced by session metadata
- replay timing will be driven by the session clock, not by the video player alone
- map, telemetry, events, and video must all align to the same session timeline

## Early Simulator Implementation Strategy

The first version of `rover-sim-next` should optimize for prompt-driven progress.

That means:
- Codex-generated rover geometry is acceptable
- rough physics values generated and tuned in code are acceptable
- generated or procedural environment assets are acceptable
- fast iteration is more important than final simulator-platform purity

Important constraint:
- AI-generated visual meshes are acceptable early
- AI-generated collision meshes and inertial properties must be validated manually
- until authoritative assets arrive, rover physics should prefer manually defined collision primitives and explicit mass/inertia values over inferred mesh physics

The key constraint is that these choices must remain replaceable behind stable contracts:
- stable MQTT contract
- stable normalized telemetry model
- stable session logging model
- stable GCS replay model

## Long-Term Simulator Evolution Path

The current baseline target for the successor simulator is now `ROS 2 + Gazebo`.

That choice fits the intended end state:
- a simulator that can substitute for the real rover during development
- clearer migration toward robotics-oriented workflows
- better long-term fit for formal robot description and world composition

Planning rules:
- keep the MQTT/GCS-facing contract stable even while the simulator stack changes underneath
- preserve the development-asset to authoritative-asset replacement path
- treat Isaac Sim as an explicit opt-in path with hard NVIDIA dependency, not a default assumption
- keep the internal simulator implementation modular enough that components can still evolve later if needed

## Headless Mode And Recording Support

The new simulator should support headless or minimally interactive execution early in the plan.

This is needed for:
- automated contract testing
- repeatable session recording
- replay fixture generation
- non-desktop execution scenarios

Session recording must not depend on a human operator watching the simulator window.

## Decision Summary

Current planning defaults:
- new sub-project name: `rover-sim-next`
- transition mode: parallel successor
- long-term simulator baseline: `ROS 2 + Gazebo`
- GCS mode: selectable backend
- backend selection first implementation: explicit config-backed selector, with optional later UI toggle
- replay UI: separate page
- logging backend first: SQLite
- initial capture scope: telemetry, control, and events
- video recording later
- AI-generated early assets allowed
- professional CAD later replaces those assets
- default map library first: `Leaflet.js`
- default CAD tooling baseline: `FreeCAD` plus optional `OpenSCAD`, with `Blender` for mesh prep
- preferred simulation asset exchange format: `glTF`/`GLB`
- preserving current MQTT/GCS compatibility is a hard transition requirement

## Final Recommendation

The best practical strategy is:
- keep `3d-env` working
- add `rover-sim-next` beside it
- preserve the current GCS-facing contract
- add map support early
- add logging and replay as first-class architecture now
- allow prompt-generated development assets at the start
- replace them later with authoritative CAD-derived assets without changing the operator workflow or replay architecture

## Next Implementation Steps

The next concrete steps should be:
- implement simulator-side logging in `3d-env`
- add a live map panel to the main GCS dashboard using the same normalized track model as replay
- turn `rover-sim-next` from scaffold into a functioning simulator backend with contract-compatible MQTT output
- define the first authoritative robot/world asset ingestion path
- add synchronized recorded video support later without changing the replay timeline model
